Front-End · HTML · CSS · JavaScript

Ecological Footprint Calculator

Overview
A web tool that measures your real environmental impact

This project is an interactive web application that calculates a user's ecological footprint — the total environmental burden of their daily lifestyle. The user answers questions across multiple categories (transport, diet, home energy, consumer habits) and the app computes a score expressed in "number of Earths" needed if everyone lived the same way. The result is displayed with contextual feedback to encourage behaviour change.

Challenge
Translating real environmental data into accurate JavaScript calculations

The core difficulty was the JavaScript calculation logic. Ecological footprint formulas involve multiple weighted variables — each factor (weekly km by car, meat frequency, electricity source, purchases) contributes differently to the final score. Designing a formula that is both scientifically grounded and simple enough for non-experts required significant research and iterative testing of the weighting system.

Solution
Modular scoring functions with weighted factor accumulation

Each category is implemented as a separate JavaScript function returning a partial score. All partial scores are accumulated and normalised against a global average baseline to produce the final result. The UI uses a step-by-step form approach so the user is never overwhelmed. Vanilla JavaScript handles all DOM manipulation and event listening — no external framework was needed.

Outcome
A fully functional calculator tested on real data

The application was tested with real personal data and returned a coherent, meaningful result. This project demonstrates DOM manipulation, multi-step UI design, mathematical modelling in JavaScript, and working with weighted scoring systems — all without any framework.

Expense Calculator

Overview
A monthly budget manager with live visual breakdown

This application helps users understand and control their monthly spending. The user inputs their initial income, then logs individual expenses by category (rent, food, transport, leisure…). The app computes total spent, remaining balance, and renders a dynamic pie chart using Chart.js that updates in real time as expenses are added or removed.

Challenge
Integrating Chart.js and handling live reactive data

The main challenge was correctly implementing Chart.js as a third-party library — reading and applying its documentation, then making the chart update reactively every time the expense list changed. The key constraint was updating the chart without re-rendering it completely, which would cause a visual flash and destroy the animation.

Solution
Dataset mutation + Chart.js .update() for smooth transitions

Expenses are stored in a JavaScript array of objects {category, amount}. On each change, the chart's data.labels and data.datasets[0].data arrays are rebuilt from the expense array, then chart.update() is called — Chart.js handles the animated transition between states. A running total counter updates simultaneously via direct DOM manipulation.

Outcome
Working budget tool validated with real monthly expenses

The application was successfully used to calculate and visualise real personal monthly expenses. This project demonstrates third-party library integration, reactive data management, and data visualisation — skills directly applicable to dashboard development in client projects.

API E-commerce

🔧 Active development — Core API is functional and Postman-tested. Improvements in progress.
Overview
A production-grade RESTful API for e-commerce back-end management

This is the most technically ambitious project to date — a complete back-end REST API for an e-commerce platform. It exposes endpoints covering the full CRUD lifecycle (Create, Read, Update, Delete) for products, users and orders. Built with Node.js and Express for the server layer, and MongoDB via Mongoose for data persistence. Designed to serve as the back-end for any front-end shopping interface.

Challenge
Route architecture and Mongoose data modelling

Two main challenges emerged. First, structuring the API route architecture cleanly using the MVC pattern — separating routes, controllers and models so the codebase stays maintainable. Second, designing Mongoose schemas correctly — particularly the relationships between collections (users → orders → products) and handling MongoDB's document model, which differs significantly from relational SQL databases.

Solution
MVC architecture with Express Router + Mongoose ODM

The project follows the MVC pattern: models/ define Mongoose schemas, controllers/ handle business logic, and routes/ map HTTP verbs to controller functions via Express Router. Each resource (product, user, order) has its own dedicated route file. Mongoose's populate() method resolves cross-collection references for nested data retrieval.

Outcome
All core routes validated on Postman — improvements ongoing

All primary routes (GET, POST, PUT, DELETE) for products and users have been validated using Postman. The API returns correct JSON responses and appropriate HTTP status codes. Currently being extended with JWT authentication middleware and input validation layers.
